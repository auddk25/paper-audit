# 学术论文写作规范（最终版）

> 基于《基于区块链预言机的安全高效电力数据共享平台构建方法》（通信学报 2025）实际行文风格提取
> 适用范围：中国顶级高校计算机科学硕士学位论文（尤其适配区块链、安全、AI方向）

---

## 一、铁律（全局硬约束）

1. **去人化**：禁用"我/我们/笔者" → 一律使用"本文/本章/该方案/本节"
2. **镜像化**：摘要要点 ↔ 第1章研究内容 ↔ 第6章总结，三处条目一一对应、措辞可变但语义必须同构
3. **量化分级**：根据提升幅度选择表达策略（详见第八节），核心原则——数据越夸张，措辞越克制
4. **逻辑闭环**：任何模块/算法的描述必须包含完整四要素 = [定义 → 输入 → 处理逻辑 → 输出]
5. **句式多样**：禁止连续两句以相同句式结构开头
6. **符号严格一致**：全文符号与符号表严格对应；跨章复用同一符号时必须重新声明语义，禁止同一符号在不同章节发生语义重载
7. **段落粒度**：每段 3–8 句（综述段和技术描述段可适度延长），首句承担 topic sentence 功能（概括本段核心论点），末句承担过渡或小结功能

---

## 二、章节宏观逻辑流

| 章节 | 逻辑流 | 核心检验标准 |
|------|--------|-------------|
| 摘要 | 背景(1句) → 问题(1句) → 方案(分步骤) → 效果(量化) | 与第1章研究内容、第6章总结逐条对应 |
| 绪论 | 场景导入(数据+趋势) → 平台功能需求 → 挑战编号枚举(每条内含"现状→不足→转折") → 方案概述 → 验证路径 | 挑战数量 = 创新点数量 |
| 相关工作 | 按技术路线递进分类 → 每类内: 概述→代表作逐条→共性局限→与本文差异 → 对比表收尾 | 分类应能自然引出本文的差异化定位 |
| 方案设计(每章) | 设计动机(1段) → 系统模型(图+文) → 实体功能描述 → 流程分阶段 → 子模块具体实现 → 案例分析(可选) | 形式化定义先于方案描述 |
| 安全性分析 | 威胁枚举 → 逐威胁定理-证明(归约式或分析式) | 每个威胁都有对应定理 |
| 性能评估 | 理论分析(公式推导) → 原型系统实现 → 实验分析(分组) → 方法对比 | 每组实验覆盖现象→对比→归因 |
| 总结与展望 | 逐条总结(镜像摘要) → 不足 → 未来方向 | 未来方向不能与已解决问题矛盾 |

## 三、核心章节内部结构

### 3.1 摘要结构

**结构**：问题(1句) → 方案名(1句) → 分步方案("首先…然后…") → 验证方式(1句) → 量化效果(1句)

**模板**：

> 针对已有[领域方案]在面对[特点]时存在的[问题]等问题，提出一种[方案全称]，简称[缩写]。首先，遵循[原则]，设计一种[模型名]，通过[技术手段]，实现[效果]；然后，针对[安全风险]，设计一种[协议名]，利用[技术列表]，有效保障[安全目标]。通过理论分析和实验验证，所提方案相比其他类似方案具有显著优势，在[安全保障]的同时能够将[效率指标]至少提升了[X]%。

### 3.2 绪论结构

```
0.1 场景导入（宏观数据 + 行业需求，1-2段）
0.2 平台功能定位（角色定义 + 技术引入的必然性，1-2段）
0.3 挑战枚举（编号列表，每条挑战内含：需求→现状→不足→转折，每条1-2段）
0.4 方案概述（"针对上述问题，本文提出…"，1段）
0.5 验证路径概述（理论+原型+实验，1段）
```

**挑战枚举模板**：

> [N])[挑战名称]。鉴于[需求描述]，这就要求[目标系统]应具备[期望属性]。然而，[现有方案类别]通常采用[已有技术路线]，而[其固有局限]使[指标]大都较弱。为此，部分学者提出了[改进方案]。不过，[改进方案的残留不足]。因此，如何[目标]就成为本文第[N]个挑战。

### 3.3 方案设计章结构

```
X 章标题
  X.1 系统模型（架构图 + 实体功能逐条描述 + 通信方式 + 流程分阶段）
  X.2 子模块设计
    X.2.1 具体实现（按模块编号描述，含伪代码）
    X.2.2 案例分析（用具体场景实例化整个流程）
```

### 3.4 安全性分析章结构

```
X 安全性分析
  开头1段：概述本节目标（证明可抵御哪些攻击）
  定理 X.1 + 证明（攻击类型1 → 归约/分析）
  定理 X.2 + 证明（攻击类型2 → 归约/分析）
  ...
```

**证明模板**：

> 定理 X　[方案名] 可抵御 [攻击类型]，[安全效果]。
> 证明　攻击者只有破解了[底层密码原语]才能[达成攻击目标]，因此只需证明[密码原语]无法被破解就可说明[方案名]可抵御[攻击]。为此，首先引入[参数定义]，然后给出[困难问题]的定义，最后描述详细的安全性分析过程。
> [归约步骤]
> 证毕。

**约束**：
- 每个在威胁模型中声明的安全目标，必须有对应的定理和证明
- 证明中使用的符号必须与符号表一致
- 如采用非形式化的安全分析，必须显式声明分析方法的局限性

### 3.5 性能评估章结构

```
X 性能评估
  X.1 理论分析（推导性能指标的解析表达式）
  X.2 系统实现（原型系统开发框架，分层描述）
  X.3 实验分析
    实验环境说明（1段）
    第一组实验：[自变量] 对 [因变量] 的影响
    第二组实验：...（与 Baseline 对比）
  X.4 方法比较（与经典方案的综合对比）
```

### 3.6 过渡方式

采用短语级自然衔接，不使用刚性整句模板：

| 过渡场景 | 推荐用法 |
|---------|---------|
| 引出需求 | "鉴于[特点]，这就要求[系统]应具备[属性]" |
| 技术必然性 | "考虑到[约束]，引入[技术]来[目的]就成为必然选择" |
| 方案引出 | "针对上述问题，本文提出一种[方案名]——[缩写]" |
| 分步设计 | "首先，…；然后，…" |
| 承上启下 | "基于此，…" / "为此，…" / "基于上述分析，…" |
| 风险引入 | "考虑到当前复杂的网络环境，以营利为目的的第三方的引入会使…存在以下安全风险。" |
| 实验引出 | "鉴于[参数]取决于[因素]，第X组实验用于说明随着[自变量]的增多[因变量]的变化情况" |
| 对比引出 | "为了体现[方案名]的优越性，本节将其与[方案列表]进行性能比较" |

## 四、问题形式化

问题形式化可前置到绪论/相关工作章，使后续讨论建立在形式化基础之上。

### 4.1 形式化定义

采用"定义X"格式，逐条给出核心概念的数学定义：

> 定义 X　[概念名] 被定义为 [N元组/函数/集合] $符号 = \{...\}$，具体含义如下。
> [逐字段解释，每个字段独立一行]

### 4.2 需求推导

从形式化定义出发，推导出系统应满足的性质：

> 根据式(X)，为了保证 $F$ 成功率，[系统]应同时保证 $f_1$ 和 $f_2$ 高效地执行。然而，考虑到[约束条件]，构建[系统]需满足以下要求。
> 1)高可靠性。...
> 2)高可扩展性。...
> 3)强安全性。...

## 五、相关工作专项约束

### 5.1 分类方式：技术路线递进

按系统架构的技术演进路线分类，每类内部遵循递进逻辑：基础方案 → 改进方案 → 仍存在的不足 → 引出下一类。

适用条件：当创新不是在某单一维度上的突破，而是在架构层面的范式升级时，按技术路线递进分类更自然。如果创新集中在特定维度（如效率+隐私），也可按创新点维度分类。

### 5.2 单类文献评述结构

```
[该类方法的产生背景/动机]
→ "为此，学者们提出一系列[目标]的共享方案。例如，"
→ [文献1]提出了[方案]，通过[技术]来实现[效果]；
→ [文献2]设计了[方案]，通过[技术]来[效果]；
→ ...（连续枚举，用分号连接）
→ [段落结尾] "虽然[该类方法]具备[优点]等特点，但是存在以下[N]个难以避免的缺陷：[缺陷1]；[缺陷2]。"
```

### 5.3 对比表

在相关工作末尾放置功能对比表，最后一行为"本文方案"，用"√/×"标注各项功能支持情况。

### 5.4 禁止行为

- 禁止纯堆砌文献而不提炼共性
- 禁止对所有文献使用相同的评价模板（"该方案存在效率不足的问题"不能出现超过两次）

## 六、实验设计与分析专项约束

### 6.1 实验设计前置说明

实验环境信息在实验分析节开头用1段文字交代，包含：
- 机器参数（OS、CPU、RAM）
- 数据集说明（来源、生成方式、统计特征）
- 通用性声明（方案适用范围与局限）

### 6.2 实验分组引出句式

每组实验以"鉴于…，第X组实验…"格式引出：

> 鉴于[因变量]取决于[影响因素]，第[N]组实验[测试了/用于说明]随着[自变量]的[变化方向][因变量]的变化情况[，并与[Baseline]进行了比较]，结果如图X所示。

### 6.3 实验分析三层结构

每组实验分析覆盖：

1. **现象层**（是什么）："从图X中不难看出：…" / "实验结果如图X所示。…"
2. **对比层**（差多少）："相比于其他方法，本文方法的整体性能明显占优。具体来说，当[条件]时，[指标]仅为[数据]。"
3. **归因层**（为什么）：轻量归因为主（1-2句），深度归因仅在方法对比节使用。
   - 轻量："此外，[技术A]无须额外优化，可直接支持[特性B]。"
   - 深度（方法对比节）："究其原因，本文方案采用了[具体技术]，避免了[baseline的瓶颈操作]，从而在[指标]上表现更优。"

### 6.4 方法对比分析句式

按方案逐个分析，每个方案1-2句：

> [方案名][因为原因]，因此[性能表现]，但[局限性]。

### 6.5 必选实验类型

| 实验类型 | 何时必做 | 目的 |
|----------|---------|------|
| 性能对比 | 始终 | 证明优于现有方案 |
| 可扩展性 | 系统涉及节点/数据规模变化 | 证明大规模场景下仍可用 |
| 参数敏感性 | 方案含可调参数 | 证明参数选取鲁棒 |
| 消融实验 | 方案含多个独立优化模块 | 证明每个模块都有贡献 |
| 安全验证 | 涉及安全性质 | 证明攻击场景下方案安全 |

### 6.6 Baseline 选取

必须论证为何选取这些对比方案（而非其他），列出对比方案并说明选取合理性。

## 七、词汇与表达约束

### 7.1 词汇替换表

| 词汇 | 处理规则 |
|------|---------|
| 用/使用 | 优先用"采用/引入/利用/借助"；技术操作描述中允许"使用"（如"使用公钥验证签名"） |
| 解决 | 替换为"规避/缓解/克服/消除/应对" |
| 但是 | 相关工作转折句中允许（如"虽然…但是存在…"）；其他位置替换为"然而/不过/遗憾的是" |
| 所以 | 替换为"因此/从而/鉴于此/由此/基于此/进而" |
| 很多/大量 | 替换为具体数据或"若干/多数/规模庞大" |
| 很快/很慢 | 替换为具体数据或"显著降低/提升" |
| 我们 | 替换为"本文/本章/本节/该方案" |
| 可以看出 | 替换为"不难看出/由图X可知/实验结果表明" |

### 7.2 连接词分类表

| 语义类型 | 可选词汇 |
|----------|---------|
| 条件引出 | 鉴于、考虑到、当进一步考虑到 |
| 必然性 | 就成为必然选择、就成为本文第X个挑战 |
| 方案引出 | 针对上述问题、为了[目标]、基于此、基于上述分析 |
| 分步 | 首先…然后…、前者…后者… |
| 递进 | 此外、不仅…还…、既能…又能…、进一步地、不仅如此、更重要的是 |
| 让步转折 | 虽然…但是…、然而…、不过…、遗憾的是、尽管如此 |
| 解释 | 也就是说、具体来说、即 |
| 因果 | 因此、从而、鉴于此、导致、这使得 |
| 归因 | 究其原因、这主要得益于、其根本原因在于、这既能…又能… |
| 举例 | 例如、以……为例、具体而言 |
| 总结 | 综上所述、由此可见、概括而言 |

### 7.3 实验对比表达约束

| 禁用 | 替换 |
|------|------|
| 急剧飙升 | 呈线性/指数增长 |
| 是XX的N倍 | 显著高于/优于 |
| 碾压 | 表现更优 |
| 完全无法满足 | 可扩展性受限 |
| 效果惊人 | 性能提升明显 |
| 几乎没有 | 开销可忽略不计 |

### 7.4 实体功能描述句式

每个系统实体用独立段落描述，首句格式统一：

> [实体名]由[维护者]来[搭建和]维护，主要负责[核心功能]。[功能展开，2-3句]。

### 7.5 流程描述句式

将整个流程分为"阶段→步骤"两级：

> 根据式(X)，整个[流程名]应分为以下[N]个阶段。
> 阶段1：[阶段名]（包含步骤①~步骤④）。[描述]
> 阶段2：[阶段名]（包含步骤⑤~步骤⑧）。[描述]

协议设计中的步骤描述：

> 步骤 1：[步骤名]。就[技术]来说，首先...然后...最后...
> 步骤 2：[步骤名]。[主语]部署[组件]，...[主语]将[组件]部署到[系统]上...

### 7.6 创新点标准句式

**通用公式**：

> **针对**[痛点]，**提出**[方案名称]，**通过**[核心技术手段]，**实现**[可量化效果]。

绪论中使用编号格式：

> **研究内容一**：[标题]。针对[痛点]，提出[方案]，通过[手段]，实现[效果]。

## 八、量化表达分级策略

**核心原则**：提升幅度越大，措辞越克制。原因——超常规的数据改进容易引发审稿人对实验公平性的质疑，用保守表达降低此风险。

| 提升幅度 | 推荐表达策略 | 示例 |
|----------|-------------|------|
| ≤15% | 直接写精确数据 | "共享效率至少提升了15%" |
| 15–30% | 写数据，可附带定性词 | "通信复杂度降低约25%，具有一定优势" |
| 30–50% | 数据为主，轻度模糊 | "存储开销降低30%以上" |
| 50–100% | 以定性描述为主，数据为辅 | "效率有明显提升（约降低60%）" |
| 1–2倍 | 弱化表达 | "表现出更优的计算性能" |
| >2倍 | 强调复杂度等级差异 | "计算复杂度由 $O(n^2)$ 降至 $O(n\log n)$" |
| >10倍 | 仅描述量级差异 | "在大规模场景下展现出量级优势" |

## 九、格式与引用规范

### 9.1 图表引用

- **先文后图**：正文先描述，图表紧随其后
- **引用句式**：
  - "如图X所示" / "实验结果如图X所示"
  - "从图X中不难看出"（分析时使用）
  - "图X举例说明了详细过程"
  - "表X给出了本文方案与现有方案的功能对比"
- 每张图表在正文中必须被至少引用一次
- 图表标题需自含义（不看正文也能理解图表内容）

### 9.2 公式引用

- 硕士论文格式：式(X-Y)，其中X为章号，Y为章内序号
- 期刊格式：式(X)，全文顺序编号
- 关键公式后必须紧跟符号说明："其中，$x$ 表示……，$y$ 表示……"
- 关键公式必须在正文中有文字解释，不能仅列出公式

### 9.3 定义引用

- 格式："如定义X所示" / "定义X"
- 定义后逐字段解释，格式可用独立行或段内列举

### 9.4 章节指代

- 大章：第X章
- 小节：X.Y节
- 跨章引用：第X章X.Y节

### 9.5 术语规范

- 英文术语首次出现时标注全称及缩写：
  - "椭圆曲线数字签名算法（ECDSA）"
  - "CKKS（Cheon-Kim-Kim-Song）同态加密"
  - "预言机服务提供商（OSP, oracle service provider）"
- 后续统一使用缩写
- 全文保持中英文术语一致性（不能一处写"区块链"另一处写"blockchain"）

## 十、案例分析/讨论（可选节）

在性能评估之后、结束语之前，可设置"案例讨论"节，用于：
1. 展示方案在真实场景中的部署架构（分层图）
2. 以具体数据请求为例，端到端走通整个流程
3. 与传统方案对比，总结优势

**优势总结格式**：

> 相较于传统基于[对比方案]的模式，基于[本文方案]的平台具有如下优点。
> 1) [优点1名]。借助[技术]的[特性]，...

## 十一、写作自检清单

### 全局检查

- [ ] **去人化**：全文无"我/我们/笔者"
- [ ] **镜像对应**：摘要 ↔ 绪论研究内容 ↔ 总结，逐条对照
- [ ] **符号一致**：符号表完整，无跨章语义重载
- [ ] **术语一致**：同一概念全文同一表述
- [ ] **段落粒度**：每段 3–8 句，首句为 topic sentence

### 章节检查

- [ ] **过渡衔接**：章节间有自然过渡短语衔接
- [ ] **创新点句式**：采用"针对…提出…通过…实现…"结构
- [ ] **相关工作分类**：分类方式能自然引出本文差异化定位
- [ ] **对比表**：相关工作末尾有功能对比表，含本文方案
- [ ] **系统模型**：含实体功能描述 + 架构图 + 流程分阶段
- [ ] **安全目标闭环**：威胁模型中每个安全目标都有对应证明

### 实验检查

- [ ] **Baseline 合理性**：已论证为何选取这些对比方案
- [ ] **三层分析**：每组实验覆盖现象→对比→归因
- [ ] **必选实验**：性能对比 + 可扩展性/敏感性/消融（按需）
- [ ] **量化分级**：提升幅度与表达策略匹配
- [ ] **图表引用前置**：无孤立图表

### 表达检查

- [ ] **禁用词清零**：无"解决/所以/很多"等禁用词（允许的例外见7.1）
- [ ] **句式多样**：无连续相同句式开头
- [ ] **连接词串联**：段落间有逻辑连接词衔接
- [ ] **归因完整**：实验现象均有技术层面解释

## 附录A：常见扣分点速查

| 问题 | 扣分原因 | 修复方法 |
|------|---------|---------|
| 摘要写了方法但没写效果 | 镜像不完整 | 补充量化结果 |
| 相关工作读完不知道gap在哪 | 分类轴与创新点脱节 | 按技术路线递进或创新维度重新分类 |
| 方案设计图看不懂 | 缺少文字配合 | 先文后图，图文对应 |
| 实验只有表没有分析 | 缺归因层 | 每组数据补归因句 |
| 符号前后矛盾 | 跨章复用未声明 | 统一符号表 + 章内重声明 |
| 安全分析只有一段话 | 缺形式化证明 | 补定理-证明结构 |
| 全文口语化 | 禁用词未替换 | 逐条对照替换表 |
| 结论与摘要内容不一致 | 镜像规则未执行 | 逐条对齐 |

## 附录B：句式速查表

| 场景 | 句式 |
|------|------|
| 数据切入 | "截至[年月]，[指标]已超过[数据]，按[统计口径]，[衍生指标]可达[数据]。" |

| 实体描述 | "[实体名]由[维护者]来维护，主要负责[功能]。" |
| 假设声明 | "本文假设[实体]是[信任属性]的。" / "本文假设[实体A]和[实体B]均为可信方。" |
| 设计动机 | "考虑到[环境/威胁]，以[目的]的第三方——[实体]的引入会使[系统]存在以下安全风险。" |
| 定理声明 | "[方案名]可抵御[攻击类型]，[防御效果]。" |
| 证明开头 | "攻击者只有破解了[密码原语]才能[攻击目标]，因此只需证明[原语]无法被破解就可说明…" |
| 实验引出 | "鉴于[因变量]取决于[因素]，第[N]组实验[测试了/用于说明]…" |
| 图表分析 | "从图X中不难看出：随着[自变量]的增加，[因变量]呈[趋势]；当[条件]时，[指标]约为[数据]。" |
| 对比分析 | "[方案A][技术原因]，因此[性能表现]，但[局限]。" |
| 量化结论 | "相较于[对比方案]，效率至少提升[X]%。" |
| 综合总结 | "综上所述，[方案名]在不同实验设置下均展现出优异且稳定的性能，具备良好的[属性]。" |
