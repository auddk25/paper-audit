# 学术论文写作规范 v3.0（Powsharing 风格修订版）

> 基于《基于区块链预言机的安全高效电力数据共享平台构建方法》（通信学报 2025）实际行文风格提取与修订
> 适用范围：中国顶级高校计算机科学硕士学位论文（尤其适配区块链、安全、AI方向）

---

## 一、铁律（全局硬约束）

1. **去人化**：禁用"我/我们/笔者" → 一律使用"本文/本章/该方案/本节"
2. **镜像化**：摘要要点 ↔ 第1章研究内容 ↔ 第6章总结，三处条目一一对应、措辞可变但语义必须同构
3. **量化分级**：根据提升幅度选择表达策略（详见第八节），核心原则——数据越夸张，措辞越克制
4. **逻辑闭环**：任何模块/算法的描述必须包含完整四要素 = [定义 → 输入 → 处理逻辑 → 输出]
5. **句式多样**：禁止连续两句以相同句式结构开头
6. **符号严格一致**：全文符号与符号表严格对应；跨章复用同一符号时必须重新声明语义
7. **段落粒度**：每段 3–8 句（允许综述段和技术描述段适度延长），首句承担 topic sentence 功能，末句承担过渡或小结功能

> **与 v2.0 的关键差异**：段落上限由6句放宽至8句，匹配期刊论文中综述段和技术描述段的实际长度需求。

## 二、章节宏观逻辑流

| 章节 | 逻辑流 | 核心检验标准 |
|------|--------|-------------|
| 摘要 | 背景(1句) → 问题(1句) → 方案(分步骤) → 效果(量化) | 与第1章研究内容、第6章总结逐条对应 |
| 绪论 | 场景导入(数据+趋势) → 平台功能需求 → 挑战编号枚举(每条内含"现状→不足→转折") → 方案概述 → 验证路径 | 挑战数量 = 创新点数量 |
| 相关工作 | 按技术路线递进分类 → 每类内: 概述→代表作逐条→共性局限→与本文差异 → 对比表收尾 | 分类应能自然引出本文的差异化定位 |
| 方案设计(每章) | 设计动机(1段) → 系统模型(图+文) → 实体功能描述 → 流程分阶段 → 子模块具体实现 → 案例分析(可选) | 形式化定义先于方案描述 |
| 安全性分析 | 威胁枚举 → 逐威胁定理-证明(归约式或分析式) | 每个威胁都有对应定理 |
| 性能评估 | 理论分析(公式推导) → 原型系统实现 → 实验分析(分组) → 方法对比 | 每组实验覆盖现象→对比→归因 |
| 总结与展望 | 逐条总结(镜像摘要) → 未来方向 | 未来方向不能与已解决问题矛盾 |

> **与 v2.0 的关键差异**：
> - 绪论逻辑改为"挑战编号枚举"模式，每条挑战内部嵌套"现状→不足→转折"微结构，而非 v2.0 的宽泛"起承转合"
> - 相关工作按"技术路线递进"分类（如：云服务→区块链→区块链预言机），而非硬性要求按创新点维度对齐
> - 新增"案例分析"和"原型系统实现"作为可选/必选节

## 三、核心章节内部结构

### 3.1 绪论结构（Powsharing 风格）

```
0.1 场景导入（宏观数据 + 行业需求，1-2段）
0.2 平台功能定位（角色定义 + 区块链预言机引入的必然性，1-2段）
0.3 挑战枚举（编号列表，每条挑战内含：需求→现状→不足→转折，每条1-2段）
0.4 方案概述（"针对上述问题，本文提出…"，1段）
0.5 验证路径概述（理论+原型+实验，1段）
```

**典型挑战枚举写法**（从 PDF 提取）：

> 1)可扩展性不足。鉴于共享请求的多样性以及智能电表安装规模的不断增加，这就要求电力数据共享平台应具备强扩展性，保证系统性能不会因请求类型增多或数据规模增大而下降。然而，当前区块链共享平台通常采用基于集中式预言机的系统架构，而有限的预言机容量使系统的性能扩展力大都较弱。为此，部分学者提出了基于分布式预言机的数据共享系统架构。不过，这些架构往往采用基于拷贝的水平扩展模式，无法应对逐渐增加的共享请求类型，导致功能扩展力低，进而增加部署和维护成本。因此，如何保证共享平台的高可扩展性就成为本文第一个挑战。

> **模板**：[挑战编号])[挑战名称]。鉴于[需求描述]，这就要求[目标系统]应具备[期望属性]。然而，[现有方案类别]通常采用[已有技术路线]，而[其固有局限]使[指标]大都较弱。为此，部分学者提出了[改进方案]。不过，[改进方案的残留不足]。因此，如何[目标]就成为本文第[N]个挑战。

### 3.2 方案设计章结构（Powsharing 风格）

```
X 章标题（如"扩展性增强的电力数据共享平台构建方法"）
  X.1 系统模型（架构图 + 实体功能逐条描述 + 通信方式 + 流程分阶段）
  X.2 子模块设计（如"基于垂直切分的分布式预言机设计"）
    X.2.1 具体实现（按模块编号描述，含智能合约伪代码）
    X.2.2 案例分析（用具体场景实例化整个流程）
```

> **与 v2.0 的关键差异**：
> - 不强制要求独立的"X.2 系统模型与问题形式化"小节；问题形式化可前置到绪论/相关工作章
> - 新增"案例分析"子节，以具体场景（如"河北省工业和信息化厅统计用电总量"）端到端走通流程
> - 实体功能描述采用"逐实体段落"格式，每段首句点明实体名称及其维护者

### 3.3 安全性分析章结构（Powsharing 风格）

```
X 安全性分析
  开头1段：概述本节目标（证明可抵御哪些攻击）
  定理 X.1 + 证明（隐私泄露 → Ring-LWE 归约）
  定理 X.2 + 证明（降质攻击 → 概率论分析）
  定理 X.3 + 证明（假冒攻击 → ECDLP 归约）
```

**证明风格**（从 PDF 提取）：

> 定理 1　Powsharing 可抵御 OSP 的窃听行为，避免隐私信息泄露。
> 证明　攻击者只有破解了CKKS全同态加密算法才能窃取Powsharing的隐私，因此只需证明CKKS算法无法被破解就可说明Powsharing可抵御隐私窃取攻击。为此，首先引入一些具体的参数和数学定义，然后给出Ring-LWE问题的定义，最后描述详细的安全性分析过程。
> [参数定义 → 困难问题定义 → 假设攻击者A → 构造归约算法R → 步骤1234 → 矛盾 → 证毕]

**模板**：

> 定理 X　[方案名] 可抵御 [攻击类型]，[安全效果]。
> 证明　攻击者只有破解了[底层密码原语]才能[达成攻击目标]，因此只需证明[密码原语]无法被破解就可说明[方案名]可抵御[攻击]。为此，首先引入[参数定义]，然后给出[困难问题]的定义，最后描述详细的安全性分析过程。
> [归约步骤]
> 证毕。

### 3.4 性能评估章结构（Powsharing 风格）

```
X 性能评估
  X.1 理论分析（推导 DSP/DSF 等指标的解析表达式）
  X.2 系统实现（原型系统开发框架，分层描述）
  X.3 实验分析
    实验环境说明（1段）
    第一组实验：[自变量] 对 [因变量] 的影响
    第二组实验：...（与 Baseline 对比）
    ...
  X.4 方法比较（与经典方案的综合对比）
```

> **与 v2.0 的关键差异**：
> - 新增"理论分析"节（公式推导性能指标），与实验分析互补
> - 新增"系统实现"节（描述原型系统技术栈），强化工程落地感
> - 实验按"组"而非按"类型"组织，每组实验独立描述自变量和观测指标

### 3.5 过渡方式

v2.0 要求刚性过渡句模板，Powsharing 风格采用更灵活的**短语级过渡**：

| 过渡场景 | Powsharing 实际用法 |
|---------|-------------------|
| 引出需求 | "鉴于[特点]，这就要求[系统]应具备[属性]" |
| 技术必然性 | "考虑到[约束]，引入[技术]来[目的]就成为必然选择" |
| 方案引出 | "针对上述问题，本文提出一种[方案名]——[缩写]" |
| 分步设计 | "首先，…；然后，…" |
| 承上启下 | "基于此，…" / "为此，…" / "基于上述分析，…" |
| 风险引入 | "考虑到当前复杂的网络环境，以营利为目的的第三方——OSP的引入会使…存在以下安全风险。" |
| 实验引出 | "鉴于[参数]取决于[因素]，第X组实验用于说明随着[自变量]的增多[因变量]的变化情况" |
| 对比引出 | "为了体现[方案名]的优越性，本节将其与[方案列表]进行性能比较" |

> **核心差异**：不使用 v2.0 的整句模板（如"第X章通过…实现了…然而…为此…本章结构安排如下…"），而是在段落中自然嵌入过渡短语。

## 五、相关工作专项约束（Powsharing 风格）

### 5.1 分类方式：技术路线递进

Powsharing 按系统架构的技术演进路线分类，而非按创新点维度分类：

```
1.2.1 基于云服务的中心化数据共享
1.2.2 基于区块链的去中心化数据共享
1.2.3 基于区块链预言机的去中心化数据共享
```

每类内部遵循递进逻辑：基础方案 → 改进方案 → 仍存在的不足 → 引出下一类。

> **适用条件**：当本文的创新不是在某单一维度上的突破，而是在架构层面的范式升级时，按技术路线递进分类更自然。

### 5.2 单类文献评述结构（从 PDF 提取）

```
[该类方法的产生背景/动机] 
→ "为此，学者们提出一系列[目标]的共享方案。例如，"
→ [文献1]提出了[方案]，通过[技术]来实现[效果]；
→ [文献2]设计了[方案]，通过[技术]来[效果]；
→ ...（连续枚举，用分号连接）
→ [段落结尾] "虽然[该类方法]具备[优点]等特点，但是存在以下[N]个难以避免的缺陷：[缺陷1]；[缺陷2]。"
```

**典型句式**（从 PDF 提取）：

> 虽然中心化数据共享具备易于实现、运行效率高、方便扩展等特点，但是存在以下2个难以避免的缺陷：单点故障，一旦云共享平台发生故障或遭到攻击，整个系统将面临瘫痪的风险；数据泄露和篡改问题，提供方一旦将数据管控权托付给CSP，无论采用何种密码保护技术，都难以彻底根除CSP对数据造成的潜在风险。

## 六、实验设计与分析专项约束（Powsharing 风格）

### 6.1 实验设计前置说明

实验环境信息在实验分析节开头用1段文字交代（而非表格），包含：
- 机器参数（OS、CPU、RAM）
- 数据集说明（来源、生成方式、统计特征）
- 通用性声明（方案适用范围与局限）

**典型写法**（从 PDF 提取）：

> 用于搭建 Powsharing 原型系统的实验机器参数为 Windows 11 23H2、Intel i7-10870H (2.2 GHz)和 16 GB RAM。本文实验中涉及的测试数据集参考典型电力数据格式（如电压、电流、功率、用电量等），通过程序随机生成具有相似统计特征的模拟数据。需要说明的是，本文方案虽然聚焦于电力数据共享场景，但对其他以数值型数据为主的应用领域具有较好的通用性，而对于非数值型的共享场景，由于无法进行有效的数据聚合操作，其通用性相对有限。

### 6.2 实验分组引出句式

每组实验以"鉴于…，第X组实验…"格式引出：

> 鉴于影响智能合约运行时间 $T_{SC}$ 的主要因素是数据共享平台中区块链的共识周期，第一组实验测试了不同共识算法和不同共享平台成员规模下共识周期的变化情况，实验结果如图7所示。

**模板**：

> 鉴于[因变量]取决于[影响因素]，第[N]组实验[测试了/用于说明]随着[自变量]的[变化方向][因变量]的变化情况[，并与[Baseline]进行了比较]，结果如图X所示。

### 6.3 实验分析结构（三层，轻归因）

每组实验分析覆盖：

1. **现象层**："从图X中不难看出：…" / "实验结果如图X所示。…"
2. **对比层**："相比于其他方法，本文方法的整体性能明显占优。具体来说，当[条件]时，[指标]仅为[数据]。"
3. **归因层**（轻量化）："此外，[技术A]无须额外优化，可直接支持[特性B]。" / "这证明了[机制]在[方面]的有效性。"

> **与 v2.0 的关键差异**：归因层不要求每组实验都写深度归因段落（"究其原因…"），允许用1-2句轻量归因。深度归因仅在方法对比节中使用。

### 6.4 方法对比节的分析句式（从 PDF 提取）

> 从图12中不难看出：Powsharing 与 MPCsharing 的性能差别不大；Hrzsharing 性能优于 PowSharing 和 MPCsharing，但其性能不够稳定；Blksharing 必须先将所有源数据上链后才能完成分享，因此消耗时间最多；Consharing 无须中间处理过程，因此所需时间较少，但存在单点故障问题；SGXsharing 在聚合运算时可以在安全区中将密文解密后，通过明文聚合再将聚合结果加密，聚合时间大大降低，因此所需时间最少，但成本相对较高。

**模板**：按方案逐个分析，每个方案1-2句，格式为"[方案名][因为原因]，因此[性能表现]，但[局限性]"。

## 七、词汇与表达约束（Powsharing 修订版）

### 7.1 词汇替换表（灵活度调整）

| 词汇 | 处理规则 | 说明 |
|------|---------|------|
| 用/使用 | **柔性替换**：优先用"采用/引入/利用/借助"，但在技术操作描述中允许"使用"（如"使用公钥验证签名"） | v2.0 硬禁止，此处放宽 |
| 解决 | 替换为"规避/缓解/克服/消除/应对" | 保持 |
| 但是 | **相关工作转折句中允许**（如"虽然…但是存在…"）；其他位置替换为"然而/不过/遗憾的是" | v2.0 硬禁止，此处放宽 |
| 所以 | 替换为"因此/从而/鉴于此/由此/基于此/进而" | 保持 |
| 很多/大量 | 替换为具体数据或"若干/多数/规模庞大" | 保持 |
| 很快/很慢 | 替换为具体数据或"显著降低/提升" | 保持 |
| 我们 | 替换为"本文/本章/本节/该方案" | 保持 |
| 可以看出 | 替换为"不难看出/由图X可知/实验结果表明" | **新增"不难看出"**（PDF 高频使用） |

### 7.2 Powsharing 高频过渡词库

| 语义类型 | Powsharing 实际高频用法 |
|----------|---------------------|
| 条件引出 | 鉴于、考虑到、当进一步考虑到 |
| 必然性 | 就成为必然选择、就成为本文第X个挑战 |
| 方案引出 | 针对上述问题、为了[目标]、基于此、基于上述分析 |
| 分步 | 首先…然后…、前者…后者… |
| 递进 | 此外、不仅…还…、既能…又能… |
| 让步转折 | 虽然…但是…、然而…、不过… |
| 解释 | 也就是说、具体来说、即 |
| 原因 | 原因如下、具体原因如下 |
| 归因 | 这既能…又能…、这主要得益于… |

### 7.3 实体功能描述句式（Powsharing 特色）

每个系统实体用独立段落描述，首句格式统一：

> [实体名]由[维护者]来[搭建和]维护，主要负责[核心功能]。[功能展开，2-3句]。

典型示例：

> 监听预言机由预言机服务提供商来维护，主要负责实时检查区块链上被数据共享合约调用的监听预言机合约状态。如果该数据共享合约与自身提供商相关，那么就生成与之对应的共享任务，并将该任务发送给提取预言机。

> 聚合预言机由预言机服务提供商来维护，主要负责面向源电力数据的聚合计算，并将计算结果转发给上链预言机。计算类型由共享请求类型来决定，包括累加、均值、方差和去除均方差大于指定倍数的异常值等。

### 7.4 流程描述句式（Powsharing 特色）

将整个流程分为"阶段→步骤"两级：

> 根据式(1)，整个共享流程应分为以下2个阶段。
> 阶段1：电力数据获取（包含步骤①~步骤④）。[描述]
> 阶段2：结果数据上传（包含步骤⑤~步骤⑧）。[描述]

协议设计中的步骤描述：

> 步骤 1：密钥分配。就 CKKS 同态加密来说，首先 UO 定义多项式环...然后...最后...
> 步骤 2：合约部署。国家电力监委会部署 LSC，...数据请求方分别将 SSC 和 USC 部署到区块链上...

## 八、量化表达分级策略（保持不变）

| 提升幅度 | 推荐表达策略 | 示例 |
|----------|-------------|------|
| ≤15% | 直接写精确数据 | "共享效率至少提升了15%" |
| 15–30% | 写数据，可附带定性词 | "通信复杂度降低约25%，具有一定优势" |
| 30–50% | 数据为主，轻度模糊 | "存储开销降低30%以上" |
| 50–100% | 以定性描述为主，数据为辅 | "效率有明显提升（约降低60%）" |
| 1–2倍 | 弱化表达 | "表现出更优的计算性能" |
| >2倍 | 强调复杂度等级差异 | "计算复杂度由 $O(n^2)$ 降至 $O(n\log n)$" |
| >10倍 | 仅描述量级差异 | "在大规模场景下展现出量级优势" |

## 九、格式与引用规范

### 9.1 图表引用

- **先文后图**：正文先描述，图表紧随其后
- **引用句式**：
  - "如图X所示" / "实验结果如图X所示"
  - "从图X中不难看出"（分析时使用）
  - "图X举例说明了详细过程"
  - "表X给出了本文方案与现有方案的功能对比"
- 每张图表在正文中必须被至少引用一次

### 9.2 公式引用

- 期刊格式：式(X)，其中X为全文顺序编号
- 硕士论文格式：式(X-Y)，其中X为章号，Y为章内序号
- 关键公式后必须紧跟符号说明："其中，$x$ 表示……，$y$ 表示……"

### 9.3 定义引用

- 格式："如定义X所示" / "定义X"
- 定义后逐字段解释，格式可用独立行或段内列举

### 9.4 术语规范

- 英文术语首次出现时标注全称及缩写，格式：
  - "椭圆曲线数字签名算法（ECDSA）"
  - "CKKS（Cheon-Kim-Kim-Song）同态加密"
  - "预言机服务提供商（OSP, oracle service provider）"
- 后续统一使用缩写
- 全文保持中英文术语一致性

## 十、摘要写作规范（Powsharing 风格）

**结构**：问题(1句) → 方案名(1句) → 分步方案("首先…然后…") → 验证方式(1句) → 量化效果(1句)

**典型写法**（从 PDF 提取）：

> 针对已有基于区块链预言机的数据共享平台在面对[特点]时存在的[问题]等问题，提出一种[方案全称]，简称[缩写]。首先，遵循[原则]，设计一种[模型名]，通过[技术手段]，实现[效果]；然后，针对[安全风险]，设计一种[协议名]，利用[技术列表]，有效保障[安全目标]。通过理论分析和实验验证，所提方案相比其他类似方案具有显著优势，在[安全保障]的同时能够将[效率指标]至少提升了[X]%。

## 十一、Powsharing 特色节：案例分析/讨论

在性能评估之后、结束语之前，可设置"案例讨论"节，用于：
1. 展示方案在真实场景中的部署架构（分层图）
2. 以具体数据请求为例，端到端走通整个流程
3. 与传统方案（如云存储）对比，总结优势

**优势总结格式**：

> 相较于传统基于云存储的数据共享模式，基于Powsharing的电力数据共享平台具有如下优点。
> 1) 共享过程可溯源。借助区块链的公开性与不可篡改性，...
> 2) 可靠数据处理：分布式预言机能够满足...
> 3) 跨域协同：通过平台统一的数据请求接口...
> 4) 隐私保护：平台仅返回结果数据而非原始信息...
