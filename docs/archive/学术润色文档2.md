# 学术论文写作规范 v2.0

> 适用范围：中国顶级高校计算机科学硕士学位论文（尤其适配区块链、安全、AI方向）

---

## 一、铁律（全局硬约束）

1. **去人化**：禁用"我/我们/笔者" → 一律使用"本文/本章/该方案/本节"
2. **镜像化**：摘要要点 ↔ 第1章研究内容 ↔ 第6章总结，三处条目一一对应、措辞可变但语义必须同构
3. **量化分级**：根据提升幅度选择表达策略（详见第八节），核心原则——数据越夸张，措辞越克制，避免审稿人质疑数据可信度
4. **逻辑闭环**：任何模块/算法的描述必须包含完整四要素 = [定义 → 输入 → 处理逻辑 → 输出]
5. **句式多样**：禁止连续两句以相同句式结构开头
6. **符号严格一致**：全文符号与符号表严格对应；跨章复用同一符号时必须重新声明语义，禁止同一符号在不同章节发生语义重载
7. **段落粒度**：每段 3–6 句，首句承担 topic sentence 功能（概括本段核心论点），末句承担过渡或小结功能

---

## 二、章节宏观逻辑流

| 章节 | 逻辑流 | 核心检验标准 |
|------|--------|-------------|
| 摘要 | 背景(1–2句) → 问题(1句) → 方案(逐条) → 效果(量化) | 与第1章研究内容、第6章总结逐条对应 |
| 绪论 | 起(宏观背景) → 承(领域具体化) → 转(痛点/Gap) → 合(本文目标+研究内容) | 痛点数量 = 创新点数量 |
| 相关工作 | 分类(维度对齐创新点) → 肯定贡献 → 指出缺陷 → 凸显本文差异 | 分类轴必须与创新点维度一一对应 |
| 方案设计(每章) | 过渡引言 → 系统模型 → 方案细节 → 安全/正确性分析 → 性能评估 → 小结 | 形式化定义先于方案描述 |
| 实验分析 | 实验设计说明 → 现象描述 → 对比差异 → 深度归因 → 消融/敏感性验证 | 每个现象必须有归因 |
| 总结与展望 | 逐条总结(镜像摘要) → 不足 → 未来方向 | 未来方向不能与已解决问题矛盾 |

## 三、核心章节内部结构

### 3.1 标准章节骨架

```
X.1 引言（过渡段）
X.2 系统模型与问题形式化
X.3 方案设计
X.4 安全性/正确性分析
X.5 性能评估
X.6 本章小结
```

### 3.2 过渡句公式

> [上章成果] + 然而/但 + [遗留问题/新需求] + 为此/鉴于此 + [本章方案概述] + [本章结构安排]

示例：
> 第3章通过引入门限签名机制实现了多方协作下的数据完整性保护。然而，该方案在节点动态加入与退出场景下的密钥更新效率仍有待优化。为此，本章提出一种基于前向安全的动态密钥管理方案。本章结构安排如下：4.2节给出系统模型与安全定义，4.3节描述方案细节，4.4节进行安全性证明，4.5节通过实验验证方案性能，4.6节总结本章工作。

### 3.3 创新点标准句式

**通用公式**：

> **针对**[痛点]，**提出**[方案名称]，**通过**[核心技术手段]，**实现**[可量化效果]。

绪论中使用编号格式：

> **研究内容一**：[标题]。针对[痛点]，提出[方案]，通过[手段]，实现[效果]。

## 四、系统模型与问题形式化（X.2节专项约束）

该节是全章的数学地基，必须包含以下四个子模块：

### 4.1 符号定义表

- 以表格形式集中列出本章所有符号及其含义
- 符号命名遵循领域惯例（如 $\lambda$ 表示安全参数，$n$ 表示节点数量）
- 若某符号在前序章节已使用但语义不同，必须显式声明："与第X章不同，本章中 $n$ 表示……"

### 4.2 系统参与方与交互模型

- 明确列出所有参与实体（如用户、服务器、矿工、验证者）
- 用图示（系统架构图）+ 文字双重描述交互关系
- 标注通信信道假设（安全信道/公开信道/认证信道）

### 4.3 安全假设与威胁模型（Threat Model）

必须回答以下三个问题：
1. **攻击者能力边界**：攻击者能做什么？（窃听、篡改、合谋、控制多少比例节点）
2. **安全假设**：方案依赖哪些计算困难性假设？（DLP、CDH、DDH、LWE 等）
3. **安全目标**：方案需要满足哪些安全性质？（机密性、不可伪造性、前向安全性等，给出形式化定义）

### 4.4 问题形式化

- 将研究问题转化为数学优化问题或判定问题
- 给出形式化的输入/输出定义
- 如适用，说明问题的计算复杂度类别

## 五、相关工作专项约束

### 5.1 分类维度对齐原则

相关工作的分类方式**必须**与本文创新点的维度对齐。

反例（错误）：创新点是"效率"和"隐私"，但相关工作按"年代"分类 → 无法凸显差异
正例（正确）：创新点是"效率"和"隐私"，相关工作分为"侧重效率的方案"和"侧重隐私的方案"两类 → 自然引出本文同时兼顾两者的定位

### 5.2 单类文献评述结构

```
[该类方法概述] → [代表性工作1的贡献] → [代表性工作2的贡献] → … 
→ [该类方法的共性局限] → [与本文方案的关键差异]
```

### 5.3 禁止行为

- 禁止纯堆砌文献而不提炼共性
- 禁止对所有文献使用相同的评价模板（"该方案存在效率不足的问题"不能出现超过两次）
- 对比表格中必须包含本文方案作为最后一行

## 六、实验设计与分析专项约束

### 6.1 实验设计前置说明（实验章节开头必写）

| 要素 | 内容 |
|------|------|
| 实验环境 | 硬件配置、操作系统、语言与库版本 |
| 数据集/参数 | 数据来源、规模、参数取值范围及依据 |
| Baseline选取 | 列出对比方案，并论证选取合理性（为什么选这几个而非其他） |
| 评估指标 | 列出所有指标，给出数学定义 |
| 实验分组 | 说明变量控制策略 |

### 6.2 实验分析三层结构

每组实验结果的分析必须覆盖：

1. **现象层**（是什么）：客观描述图表中的趋势/数据，使用"如图X.X所示"引导
2. **对比层**（差多少）：与 baseline 进行定量对比
3. **归因层**（为什么）：解释造成差异的技术原因，归因到具体的设计决策

归因句式：
> 究其原因，本文方案采用了[具体技术]，避免了[baseline的瓶颈操作]，从而在[指标]上表现更优。
> 这主要得益于[机制名称]在[场景]下的[优势]。

### 6.3 必选实验类型

| 实验类型 | 何时必做 | 目的 |
|----------|---------|------|
| 性能对比 | 始终 | 证明优于现有方案 |
| 可扩展性 | 系统涉及节点/数据规模变化 | 证明大规模场景下仍可用 |
| 参数敏感性 | 方案含可调参数 | 证明参数选取鲁棒 |
| 消融实验 | 方案含多个独立优化模块 | 证明每个模块都有贡献 |
| 安全验证 | 涉及安全性质 | 证明攻击场景下方案安全 |

## 七、词汇与表达约束

### 7.1 禁用词替换表

| 禁用 | 替换选项 |
|------|---------|
| 用/使用 | 采用、引入、利用、借助 |
| 解决 | 规避、缓解、克服、消除 |
| 但是 | 然而、但、不过、遗憾的是 |
| 所以 | 因此、从而、鉴于此、由此 |
| 很多/大量 | [具体数据]、若干、多数 |
| 很快/很慢 | [具体数据]、显著降低/提升 |
| 我们 | 本文、本章、本节、该方案 |
| 可以看出 | 由图X可知、实验结果表明 |

### 7.2 连接词分类表

| 语义类型 | 可选词汇 |
|----------|---------|
| 顺序 | 首先、其次、再次、最后 |
| 承接 | 接着、随后、与此同时、在此基础上 |
| 递进 | 此外、进一步地、不仅如此、更重要的是 |
| 转折 | 然而、遗憾的是、相比之下、尽管如此 |
| 因果 | 因此、从而、鉴于此、导致、这使得 |
| 归因 | 究其原因、这主要得益于、其根本原因在于 |
| 举例 | 例如、以……为例、具体而言 |
| 总结 | 综上所述、由此可见、概括而言 |

### 7.3 实验对比表达约束

| 禁用 | 替换 |
|------|------|
| 急剧飙升 | 呈线性/指数增长 |
| 是XX的N倍 | 显著高于/优于 |
| 碾压 | 表现更优 |
| 完全无法满足 | 可扩展性受限 |
| 效果惊人 | 性能提升明显 |
| 几乎没有 | 开销可忽略不计 |

## 八、量化表达分级策略

**核心原则**：提升幅度越大，措辞越克制。原因——超常规的数据改进容易引发审稿人对实验公平性的质疑，用保守表达降低此风险。

| 提升幅度 | 推荐表达策略 | 示例 |
|----------|-------------|------|
| ≤15% | 直接写精确数据 | "计算开销降低约12%" |
| 15–30% | 写数据，可附带定性词 | "通信复杂度降低约25%，具有一定优势" |
| 30–50% | 数据为主，轻度模糊 | "存储开销降低30%以上" |
| 50–100% | 以定性描述为主，数据为辅 | "效率有明显提升（约降低60%）" |
| 1–2倍 | 弱化表达 | "表现出更优的计算性能" |
| >2倍 | 强调复杂度等级差异 | "计算复杂度由 $O(n^2)$ 降至 $O(n\log n)$" |
| >10倍 | 仅描述量级差异 | "在大规模场景下展现出量级优势" |

## 九、格式与引用规范

### 9.1 图表引用

- **先文后图**：正文先描述，图表紧随其后
- **引用前置**：以"如图X.X所示"或"表X.X给出了……"作为引导句开头
- 每张图表在正文中必须被至少引用一次
- 图表标题需自含义（不看正文也能理解图表内容）

### 9.2 公式引用

- 格式：式(X-Y)，其中X为章号，Y为章内序号
- 关键公式必须在正文中有文字解释，不能仅列出公式
- 公式后需紧跟符号说明（"其中，$x$ 表示……，$y$ 表示……"）

### 9.3 章节指代

- 大章：第X章
- 小节：X.Y节
- 跨章引用：第X章X.Y节

### 9.4 术语规范

- 英文术语首次出现时标注全称及缩写："联邦学习（Federated Learning, FL）"
- 后续统一使用缩写
- 全文保持中英文术语一致性（不能一处写"区块链"另一处写"blockchain"）

## 十、安全性/正确性分析专项约束（X.4节）

### 10.1 证明结构

```
定理 X.X（安全性质名称）：在[假设]下，所提方案满足[安全性质]。
证明：
  （1）形式化归约目标
  （2）构造模拟器/归约算法
  （3）分析成功概率
  （4）得出结论
```

### 10.2 约束

- 每个在威胁模型中声明的安全目标，必须有对应的定理和证明
- 证明中使用的符号必须与 X.2 节的符号表一致
- 如采用非形式化的安全分析（而非严格归约证明），必须显式声明分析方法的局限性

## 十一、写作自检清单

### 全局检查

- [ ] **去人化**：全文无"我/我们/笔者"
- [ ] **镜像对应**：摘要 ↔ 绪论研究内容 ↔ 总结，逐条对照
- [ ] **符号一致**：符号表完整，无跨章语义重载
- [ ] **术语一致**：同一概念全文同一表述
- [ ] **段落粒度**：每段 3–6 句，首句为 topic sentence

### 章节检查

- [ ] **过渡句**：每章 X.1 节包含标准过渡段
- [ ] **创新点句式**：采用"针对…提出…通过…实现…"结构
- [ ] **相关工作分类轴**：与创新点维度对齐
- [ ] **系统模型四要素**：符号表 + 参与方模型 + 威胁模型 + 问题形式化
- [ ] **安全目标闭环**：威胁模型中每个安全目标都有对应证明

### 实验检查

- [ ] **Baseline 合理性**：已论证为何选取这些对比方案
- [ ] **三层分析**：每组实验覆盖现象→对比→归因
- [ ] **必选实验**：性能对比 + 可扩展性/敏感性/消融（按需）
- [ ] **量化分级**：提升幅度与表达策略匹配
- [ ] **图表引用前置**：无孤立图表

### 表达检查

- [ ] **禁用词清零**：无"使用/解决/但是/很多"等禁用词
- [ ] **句式多样**：无连续相同句式开头
- [ ] **连接词串联**：段落间有逻辑连接词衔接
- [ ] **深度归因**：实验现象均有技术层面解释

## 附录：常见扣分点速查

| 问题 | 扣分原因 | 修复方法 |
|------|---------|---------|
| 摘要写了方法但没写效果 | 镜像不完整 | 补充量化结果 |
| 相关工作读完不知道gap在哪 | 分类轴与创新点脱节 | 按创新维度重新分类 |
| 方案设计图看不懂 | 缺少文字配合 | 先文后图，图文对应 |
| 实验只有表没有分析 | 缺归因层 | 每组数据补"究其原因"段 |
| 符号前后矛盾 | 跨章复用未声明 | 统一符号表 + 章内重声明 |
| 安全分析只有一段话 | 缺形式化证明 | 补定理-证明结构 |
| 全文口语化 | 禁用词未替换 | 逐条对照替换表 |
| 结论与摘要内容不一致 | 镜像规则未执行 | 逐条对齐 |
