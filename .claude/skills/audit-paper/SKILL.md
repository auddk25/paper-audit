# 论文审计 Skill

审计东北大学硕士学位论文，检查格式规范、内容质量和学术写作风格。

## 输入

用户提供论文文件路径作为参数。如果没有提供，提示用户输入。

论文和数据文件统一放在 `E:/code/paper-audit/input/` 目录下：

```
input/
├── pdf/          ← 论文 PDF（格式检查用）
├── word/         ← 论文 Word .docx（空行检查用）
└── data/         ← 实验原始数据（计算验证用）
    ├── ch3/      ← 第3章实验数据（CSV/Excel/JSON/TXT）
    ├── ch4/      ← 第4章实验数据
    └── ch5/      ← 第5章实验数据
```

## 参考文档

**唯一权威文档**：`E:/code/paper-audit/学术写作规范.md`

此文档包含 LLM 审计依据：
- 写作风格规范（11节 + 自检清单 + 句式速查表）
- 排版格式由 Python 脚本自动检查，不在此文档中

> **约束：LLM 默认只读 `学术写作规范.md`，禁止主动读取原始参考文件。**
> 原始文件（`5-论文格式（硕士）.pdf`、`基于区块链…鲁宁.pdf`）仅在用户明确要求"更新写作规范"时才去读取提取。

## 执行流程 — 五阶段流水线

Main agent 全程控制流水线，根据阶段特性决定串行/并行。

```
┌─────────────────────────────────────────────────────────────┐
│ Stage 1: 自动化预检（全部并行）                               │
│   ├── format_checker.py    ← PDF 格式检查                    │
│   ├── word_checker.py      ← Word 空行检查                   │
│   ├── cross_ref_checker.py ← 交叉引用检查                    │
│   └── pdf_extractor.py     ← 结构提取                        │
│                                                             │
│ Gate: 等待全部脚本完成，收集 JSON 结果                         │
├─────────────────────────────────────────────────────────────┤
│ Stage 2: 通读建档（main agent 串行，不可跳过）                 │
│   ├── 读取写作规范（学术写作规范.md）                          │
│   ├── 快速通读全文 PDF（每章略读，不做详细审查）                │
│   └── 输出「论文上下文档案」：                                 │
│       ├── 每章核心论点（1-2句）                               │
│       ├── 关键术语表（术语 → 定义 → 首次出现章节）              │
│       ├── 实验数据声明清单（第N章声明了哪些数值）               │
│       └── 章节依赖关系（哪些章引用了哪些章的内容）              │
│                                                             │
│ Gate: 上下文档案就绪，main agent 决定 Stage 3 调度策略          │
├─────────────────────────────────────────────────────────────┤
│ Stage 3: 逐章深度审查（main agent 调度，混合串并行）            │
│                                                             │
│   调度规则：                                                  │
│   ├── 有跨章依赖的章节 → 串行（按依赖顺序）                    │
│   ├── 无跨章依赖的章节 → 可并行 sub-agent                     │
│   ├── 实验章节（3/4/5）→ 必须先读原始数据再审查                │
│   └── main agent 根据 Stage 2 的依赖关系图自行决策             │
│                                                             │
│   每个章节/部分的分析任务和输出文件见下方「分析单元」            │
│                                                             │
│ Gate: 所有章节报告就绪                                        │
├─────────────────────────────────────────────────────────────┤
│ Stage 4: 全局检查（main agent 串行，需要全文视野）              │
│   ├── 交叉引用合理性（基于 Stage 1 的 cross_ref 结果）         │
│   ├── 镜像对应：摘要 ↔ 绪论 ↔ 总结                           │
│   ├── 术语一致性：跨章术语对照                                 │
│   └── 输出 output/09-全局检查.md                              │
│                                                             │
│ Gate: 全局检查完成                                            │
├─────────────────────────────────────────────────────────────┤
│ Stage 5: 汇总（main agent）                                  │
│   └── 格式/内容/写作三维评分 + 重点改进建议                    │
└─────────────────────────────────────────────────────────────┘
```

### Stage 1: 自动化预检

**并行**启动以下 4 个任务：

```bash
# 任务 A: PDF 格式检查
E:/code/paper-audit/.venv/Scripts/python.exe E:/code/paper-audit/scripts/format_checker.py "<PDF路径>"

# 任务 B: Word 空行检查
E:/code/paper-audit/.venv/Scripts/python.exe E:/code/paper-audit/scripts/word_checker.py "<DOCX路径>"

# 任务 C: 交叉引用检查
E:/code/paper-audit/.venv/Scripts/python.exe E:/code/paper-audit/scripts/cross_ref_checker.py "<PDF路径>"

# 任务 D: 结构提取
E:/code/paper-audit/.venv/Scripts/python.exe -u -c "
import sys, io, json
sys.stdout = io.TextIOWrapper(sys.stdout.buffer, encoding='utf-8')
sys.path.insert(0, 'E:/code/paper-audit/scripts')
from pdf_extractor import extract_structure
print(json.dumps(extract_structure(r'<PDF路径>'), ensure_ascii=False, indent=2))
"
```

收集所有 JSON 结果后进入 Stage 2。将格式检查结果立即写入 `output/08-格式检查.md`。

### Stage 2: 通读建档

**Main agent 串行执行，不可委派 sub-agent。**

1. 读取 `E:/code/paper-audit/学术写作规范.md`
2. 使用 Read 工具快速通读全文 PDF（每次20页，略读提取关键信息）
3. 生成「论文上下文档案」（保存在 main agent 上下文中，不写文件）：

```
论文上下文档案：
├── 论文标题 + 作者
├── 章节结构（来自 Stage 1 结构提取）
├── 每章核心论点（1-2句总结）
├── 关键术语表：
│   └── 术语 | 定义/含义 | 首次出现章节
├── 实验数据声明清单：
│   └── 第N章 | 数据项 | 论文声明值 | 页码
├── 章节依赖关系：
│   └── 第N章引用了第M章的内容 → 串行
│   └── 第N章独立 → 可并行
└── 跨章引用清单（来自 Stage 1 cross_ref 结果）
```

4. **调度决策**：基于依赖关系，决定 Stage 3 的执行顺序。规则：
   - 有依赖 → 按依赖链串行
   - 无依赖 → 用 Agent 工具并行派发 sub-agent
   - 拿不准 → 宁可串行，不冒精度风险

### Stage 3: 逐章深度审查

#### 分析单元

| 部分 | 输出文件 | 并行条件 |
|------|---------|---------|
| 前置部分（封面/摘要/目录） | `output/00-前置部分.md` | 通常可并行 |
| 第1章 绪论 | `output/01-第1章.md` | 通常串行（后续章节依赖绪论） |
| 第2章 相关工作 | `output/02-第2章.md` | 依赖第1章 |
| 第3章 实验 | `output/03-第3章.md` | 需原始数据，可能依赖第2章 |
| 第4章 实验 | `output/04-第4章.md` | 需原始数据，视依赖关系定 |
| 第5章 实验 | `output/05-第5章.md` | 需原始数据，视依赖关系定 |
| 第6章 总结 | `output/06-第6章.md` | 必须串行（需所有前置章节） |
| 后置部分（参考文献/致谢） | `output/07-后置部分.md` | 通常可并行 |

#### 每个分析单元的执行步骤

无论 main agent 还是 sub-agent，每个单元都执行相同流程：

**输入**：
- 论文 PDF 对应页码范围（Read 工具，每次最多20页）
- 学术写作规范.md（首次分析时读取）
- 论文上下文档案（main agent 传递给 sub-agent 的 prompt 中包含）

**分析内容**：

1. **逻辑问题**：逻辑断裂、论证不充分、前后矛盾，精确到页码
2. **语病问题**：病句、主语缺失、搭配不当、冗余表述，给出原文+修改建议
3. **润色建议**（对照写作规范）：
   - 去人化：检测"我/我们/笔者"
   - 禁用词：对照7.1节替换表
   - 句式多样性：是否有连续相同句式开头
   - 段落粒度：每段是否3-8句
   - 量化表达：提升幅度与表达策略是否匹配
4. **内容审查**：引用规范、图表引用、章节衔接
5. **计算验证**：遇到数值计算/统计结果/公式推导 → **必须写 Python 代码验算**，绝不心算
   1. 提取论文中的计算公式、输入参数、输出结果
   2. 编写 Python 脚本复现计算
   3. 用 Bash 工具执行脚本，对比论文结果
   4. 有偏差 → 标注"计算存疑" + 论文值 vs 验算值

#### 实验章节强制约束（第3/4/5章）

> **铁律：分析实验章节时，必须先读取 `input/data/chN/` 下的原始数据文件，再审查论文内容。**

1. 用 Glob 扫描 `E:/code/paper-audit/input/data/ch{章号}/`，列出所有数据文件
2. 用 Read 读取数据文件（CSV/Excel/JSON/TXT），理解原始数据结构
3. 论文中每个实验数据（吞吐量、延迟、准确率、提升百分比等）：
   - 编写 Python 脚本从原始数据独立计算
   - 对比论文声明值 vs 原始数据计算值
   - 相对误差 > 1% → 标记"数据存疑"
4. 若 `input/data/chN/` 为空 → 报告标注"⚠ 第N章无原始数据，无法交叉验证"

#### 报告格式

每个分析单元完成后**立即**写入对应输出文件，统一结构：

```markdown
# [部分名称] 审计报告
**论文**: [标题]  **审计时间**: [时间]  **页码范围**: [起始-结束]

## 一、逻辑问题
[精确到页码]

## 二、语病问题
[原文 + 修改建议]

## 三、润色建议
[原文 + 修改建议，对照写作规范]

## 四、内容审查
[引用规范、图表引用、章节衔接]

## 五、数据验证（仅实验章节）
[论文值 vs 原始数据计算值]
```

### Stage 4: 全局检查

**Main agent 串行执行**，需要全文视野。

1. **交叉引用合理性**：Stage 1 的 cross_ref 结果中 invalid → 直接报错；valid → 审查上下文是否合理
2. **镜像对应**：摘要 ↔ 绪论研究内容 ↔ 总结章，检查是否一一对应
3. **术语一致性**：汇总全文术语，检查同一概念是否存在不同表述

输出：`output/09-全局检查.md`

---

### Stage 5: 汇总

输出最终综合评分到 `output/09-全局检查.md` 末尾：
- 格式 / 内容 / 写作 三维评分
- Top 10 重点改进建议

## Sub-agent 调度规范

当 main agent 决定用 sub-agent 时，遵守以下规则：

1. **传递完整上下文**：sub-agent 的 prompt 中必须包含：
   - 论文上下文档案（Stage 2 生成的）
   - 该章节的 PDF 路径 + 页码范围
   - 学术写作规范.md 的路径
   - 实验数据目录路径（如适用）
   - 输出文件路径
2. **不可二次派发**：sub-agent 不得再派发 sub-agent
3. **独立输出**：每个 sub-agent 直接写入自己的输出文件
4. **main agent 等待**：所有 sub-agent 完成后才进入 Stage 4
5. **精度优先**：拿不准能否并行 → 串行

## 注意事项

- 论文限110页以内，每章不超过20页
- 读取 PDF 时使用 Read 工具的 pages 参数分批读取（每次最多20页）
- 格式检查由 Python 脚本完成，LLM 不要重复检查格式问题
- 内容审查和润色要深入分析，不要泛泛而谈
- 润色建议要给出具体的原文和修改建议，精确到页码
- **铁律：LLM 不做计算**。所有涉及数值的验证必须写 Python 代码执行，用代码输出作为判断依据
